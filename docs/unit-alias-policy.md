# Unit Alias Policy

This document defines how RosettaStune maps linked-data unit identifiers (QUDT, OM, UN/CEFACT, IOF adapter aliases) to stable canonical units and backend unit libraries.

## Goals

- Deterministic local resolution with no network calls.
- One canonical unit token per semantic unit concept.
- Multiple schema aliases per canonical token.
- Consistent behavior across Rust and Python implementations.

## Source Of Truth

- Lexicon file: `python/rosettastune/data/lexicon.jsonld`
- Resolver core: `src/lib.rs`
- Python backend adapters: `python/rosettastune/api.py`

The JSON-LD graph is the portable interchange format. Other language implementations should ingest this lexicon directly and not hardcode schema-specific aliases in code.

## Canonical Token Rules

- Canonical tokens are short, explicit, and ASCII (examples: `kg`, `J/kg`, `degC`).
- Use a single canonical token for each physical concept.
- Keep canonical tokens stable once released.
- If a canonical token must change, add compatibility aliases and document migration.

## Alias Rules

- Every entry should include aliases from as many supported schemas as possible.
- Include compact forms (for example `qudt:KILO_GM`) and expanded IRI forms are generated by the resolver.
- Include IOF adapter aliases (`iof_adapter:*`) when an IOF measurement path is common in data ingestion.
- Do not include ambiguous aliases that could resolve to different dimensions.

## Dimension Rules

- Each entry should set `dimension` (for example `mass`, `pressure`, `temperature`).
- Dimension labels are for validation and diagnostics.
- Two entries with the same alias but different dimensions are not allowed.

## Backend Mapping Rules

Each canonical token should map in all available backends whenever physically meaningful.

- Pint mapping in `_pint_unit`.
- Astropy mapping in `_astropy_unit`.
- SymPy mapping in `_sympy_unit`.

If a backend cannot represent a unit safely (for example `degC` in SymPy as a linear token), raise a clear `ValueError`.

## Add A New Unit

1. Add the JSON-LD entry in `python/rosettastune/data/lexicon.jsonld`.
2. Add canonical mappings in `python/rosettastune/api.py`.
3. Add tests in `tests/test_rosettastune.py`:
   - alias resolution (`pytest.mark.parametrize`)
   - backend support behavior
   - failure behavior where needed
4. Refresh snapshots:

```bash
INSTA_UPDATE=always cargo test
```

5. Re-run full validation:

```bash
cargo fmt --check
cargo test
/var/home/wturner/-wrk/-ve311/rosettastune/bin/python -m pytest -q tests/test_rosettastune.py
```

## Compatibility And Versioning

- Additive alias updates are preferred.
- Removing aliases is a breaking change.
- Repointing an alias to a different canonical token is a breaking change.
- Keep changelog notes for any mapping behavior change.

## CI Expectations

CI should fail if:

- Rust formatting or tests fail.
- Snapshot output drifts unexpectedly.
- Python backend or adapter tests fail.

This ensures lexicon edits stay consistent across the Rust resolver and Python unit adapters.
